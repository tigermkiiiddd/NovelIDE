# NovelIDE 长篇 AI 创作痛点评估

> 记录日期: 2026-04-26
> 评估视角: 从 AI 长篇小说/长篇文章创作痛点反推 NovelIDE 当前设计范式

## 总体结论

NovelIDE 已经不是普通的 AI 写作工具，而更接近“长篇创作基础设施”。

它的核心优势不是单次生成，而是围绕长篇创作最难的三个问题建立系统能力：

- 长期一致性：记忆宫殿、项目资产、时间线、伏笔和角色状态追踪。
- 上下文工程：稳定协议、运行时状态、工作记忆、正史检索和短 transcript。
- 作者主权：可定制 Skill、可手动管理记忆、Agent 辅助整理、审批流、Diff、局部 patch、版本管理和意图分类。
- 软工作流：以 Skill 承载可编辑、可激活、可组合的创作流程，形成介于 prompt 和 hard tool calling 之间的 soft tool calling。

当前总体评分：**8.45 / 10**。

下一阶段重点不是继续堆能力，而是把以下三件事做成闭环：

- 质量评判闭环。
- 风格一致性硬约束。
- Agent 工作流执行可靠性验证。

## 分项观察

### 1. 记忆断裂

长篇 AI 创作的根问题之一是模型会忘。NovelIDE 的解法是正确的：不要依赖聊天历史当事实来源，而是把作品事实沉淀到项目资产、记忆宫殿、大纲、时间线和角色工具中。

这使系统从“聊天续写”转向“正史驱动写作”。

### 2. 结构失控

AI 很容易写出局部通顺但整体无递进的文本。NovelIDE 已经有大纲、时间线、故事线、伏笔和情绪曲线等结构工具。

更重要的是，正文写作流程和正文扩写这类 Skill 已经在做 soft workflow：它不是硬工具调用，但会把写作拆成可执行步骤、检查点和反模式。这能让模型在不增加工具复杂度的情况下，按照作者定义的流程写作。

后续关键不是新增结构工具，而是让 soft workflow 更稳定地承接结构召回：本章目标、当前故事线位置、未回收伏笔、角色状态、上一章结尾。

### 3. 注水与空转

“驱动力/目的性”双轴是非常好的质量核心。它比泛泛要求“写得精彩”更可执行。

正文扩写 Skill 进一步把抽象标准变成软执行规则，例如：

- 展示而非讲述。
- 对话必须服务角色即时目标。
- 行为必须有动机。
- 伏笔必须写进正文。
- 群众和配角不能只是背景板。

建议后续把它变成审核器：

- 每段是否有新增信息、行动、冲突、情绪变化或关系变化。
- 本段驱动力类型是什么。
- 删掉本段，读者会少获得什么。

### 4. 角色漂移

角色漂移不只是“设定忘了”，还包括语气、关系温度、决策边界、能力边界漂移。

现有角色档案和章节分析能解决“状态漂移”，但“声音漂移”还需要更明确的样本库和对比机制。

### 5. 风格不稳定

Soul 分层解决了风格继承的作用域问题，避免跨项目污染，这是很好的基础。

但风格稳定不能只靠文字规则。后续可以考虑建立 prose fingerprint：

- 句长分布。
- 对话/叙述比例。
- 心理描写密度。
- 感官描写偏好。
- 常用句式和禁用句式。
- 示例片段对齐。

### 6. 局部好、全局弱

AI 单场景生成能力强，但常常不知道这一场戏在整本书中的功能。NovelIDE 的 Context OS 范式正好对症。

soft workflow 的价值在这里很明显：它把“知道大局”推进到“按大局执行”。正文写作流程负责召回大纲、上一章、伏笔、角色状态；正文扩写负责把细纲转成有目的、有画面、有主体性的正文。

关键是每次正文写作都要回答：

- 这一章推进哪条故事线。
- 这一场改变了谁。
- 这一段承担什么目的。
- 它和前后章节的因果关系是什么。

### 7. 修改困难

长篇创作中，修改比生成更常见。NovelIDE 的 Diff、局部 patch、版本历史和审批流是很强的基础。

后续应继续强化“最小改动原则”：用户要求局部修改时，默认只改目标范围，不额外重写、不额外建档、不额外同步非必要资产。

### 8. 上下文成本高

NovelIDE 的 Context OS 是当前最有架构价值的方向。它把上下文拆成稳定协议、运行态、工作记忆、正史检索和短期 transcript，避免把一切塞进长 prompt。

这不仅省 token，更重要的是降低噪声，让模型知道哪些信息是事实、哪些是临时对话。

### 9. 作者主权被稀释

AI 太主动会抢方向盘，太被动又只是打字机。NovelIDE 的作者主权不只来自审批流和意图分类，还来自更底层的“可改造性”。

作者不仅能决定是否接受 AI 的改动，也能改造 AI 的创作方法：

- Skill 是可定制的，作者可以调整创作方法论、题材规则、审核标准和工作流。
- 记忆不是黑箱，作者可以手动管理，也可以让 Agent 辅助整理。
- Soul 分层让作者能区分跨项目偏好和当前项目特殊要求。
- Diff、版本管理和审批流保证最终进入作品的内容仍由作者确认。

因此 NovelIDE 的作者主权比普通 AI 写作器更高。普通产品常见的问题是“作者只能改 prompt”，而 NovelIDE 允许作者改 AI 的工作框架、项目正史和记忆结构。

soft workflow 进一步提高了主权上限。它让作者不只是写 prompt，也不是只能开关硬工具，而是能编辑 AI 的“执行规程”。这是一种介于 prompt 和 tool calling 之间的控制方式：作者定义流程，Agent 按流程执行，必要时再调用 hard tools。

过度反应 bug 已经证明：问题不是系统能力太强，而是能力缺少正确的 intent gate。修复方向是对的：简单意图走轻路径，复杂任务才进入完整工作流。

后续需要注意的不是削弱自动化，而是继续提高控制面的可审计性。当前输入区已经显示激活 Skill、常驻/召回记忆和活跃文档，作者能移除 Skill、隐藏常驻记忆、移除召回记忆；这说明 NovelIDE 已经把一部分“AI 正在携带什么上下文”产品化了。

下一步不是从零做控制面，而是补足更细的解释层：哪些内容会被沉淀、Agent 为什么选择某个工作流、某个 Skill 是由用户触发、关键词触发、语义触发还是 Agent 主动激活。

### 10. 质量评判缺标准

NovelIDE 已经有双轴、期待感、爽点节奏、编辑审读等标准。下一步应把这些标准从 prompt 规则变成可观察结果。

建议形成章节级诊断报告：

- 驱动力评分。
- 目的性评分。
- 注水风险。
- OOC 风险。
- 伏笔推进/回收状态。
- 结构递进状态。
- 风格偏移程度。

## 迭代方向

### 方向一：质量评判闭环

目标：让正文生成后自动进入可解释的质量检查，而不是只靠用户主观感觉。

可落地能力：

- 章节质量评分。
- 段落目的性检测。
- 注水段落标记。
- 冲突递进检测。
- 期待感断点检测。

### 方向二：风格一致性硬约束

目标：让项目风格不只存在于 Soul 和规则里，而是有样本、有指标、有对齐。

可落地能力：

- 项目 prose fingerprint。
- 角色 voice fingerprint。
- 写作前样本召回。
- 写作后风格偏移检测。

### 方向三：工作流执行可靠性验证

目标：确保 Agent 真的按照意图分类和技能边界执行，而不是只在 prompt 中承诺。

可落地能力：

- 固定测试用例：闲聊、简单指令、局部润色、单章规划、整本设计。
- 工具调用轨迹断言。
- 技能激活边界测试。
- 工具失败熔断测试。
- 用户否定后的方向放弃测试。

### 方向四：作者控制面可审计化

状态：已有基础。输入区已经有激活 Skill 显示、记忆加载显示、召回记忆显示和活跃文档显示；作者可以移除 Skill，也可以隐藏或移除部分记忆上下文。

目标：把已经存在的作者主权能力继续做深，让作者不只“看得见、改得动”，还可以理解“为什么被加载、将影响什么、是否会被沉淀”。

可落地能力：

- Skill 触发来源说明：用户指定、关键词、语义匹配、Agent 主动激活。
- Skill 编辑与版本历史。
- 本轮召回记忆的命中原因。
- 本轮将沉淀记忆预览。
- Agent 工作流选择原因摘要。
- 项目 Soul / 全局 Soul / 记忆宫殿的作用域提示。

### 方向五：Soft Workflow 工程化

状态：已有基础。正文写作流程是端到端 hard/soft 混合 workflow；正文扩写是典型 soft workflow，`tools: []`，但提供明确的触发条件、执行原则、检查步骤和反模式。

目标：把 Skill 从“可加载的长提示词”进一步稳定为 soft tool calling。它不一定执行代码，但应该像工具一样有输入、前置条件、执行步骤、产出标准和失败边界。

可落地能力：

- Skill 类型标记：`workflow`、`soft_workflow`、`style_rule`、`review_rule`。
- Soft workflow metadata：适用任务、输入要求、输出要求、检查清单、反模式。
- Workflow trace：显示本轮激活了哪些 soft workflow，以及执行到哪一步。
- Soft workflow completion check：让 Agent 在 final_answer 前报告已满足/未满足的关键检查项。
- Soft workflow 与 hard tool 的边界：没有数据读写需求时只走 soft workflow，需要正史查询/同步时再解锁 hard tools。

## 范式总结

NovelIDE 最适合坚持的范式是：

> Canon-first, soft-workflow-guided, author-approved longform writing IDE.

也就是：

- 以作品正史为中心，而不是以聊天历史为中心。
- 以可定制 soft workflow 引导 AI，而不是裸 prompt 生成。
- 以可定制 Skill、可管理记忆和作者审批保留主权，而不是让 Agent 自动接管。
- 以轻重路径分流控制复杂度，而不是所有任务都进入重流程。

这套范式的关键不是“让 AI 更主动”，而是：

> 对复杂任务足够系统，对简单任务足够克制。
